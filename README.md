# Telegram Auto Poster

`Telegram Auto Poster` (`TAP`) — это self-hosted сервис для публикации файлов из локальных папок в Telegram-каналы по расписанию. Приложение работает в одном Docker-контейнере, управляется через веб-интерфейс и не требует отдельного Redis, Postgres или внешних воркеров.

Проект подходит для сценариев, где нужно:

- подключить одну или несколько папок с контентом через Docker volumes
- индексировать большие архивы изображений, GIF, видео и документов
- публиковать файлы в один или несколько Telegram-каналов
- контролировать очередь, историю отправок, ручные запуски и предпросмотр файлов

## Быстрый старт

Текущий пример `docker-compose.yml` в репозитории:

```yaml
services:
  app:
    image: ghcr.io/eranaki/telegram-auto-poster:latest
    container_name: telegram-folder-poster
    restart: unless-stopped
    ports:
      - "1338:8080"
    environment:
      APP_DATA_DIR: /data
      APP_DB_PATH: /db/app.db
      APP_TIMEZONE: Europe/Moscow
      SCHEDULER_TICK_SECONDS: 30
      SCAN_TICK_SECONDS: 120
    volumes:
      - ./data:/data
      - ./db:/db
      - ./example-content:/content/example:ro
```

Пока используются стандартные данные входа, в интерфейсе показывается предупреждающая плашка.

## Структура данных внутри приложения

Приложение опирается на несколько основных сущностей:

- `Канал` — Telegram-канал с собственными настройками отправки
- `Источник` — папка с контентом внутри контейнера
- `Правило` — сценарий публикации с расписанием, фильтрами и параметрами отправки
- `Файл` — запись о найденном файле в индексе базы данных
- `История` — журнал попыток отправки
- `Очередь` — предпросмотр кандидатов на публикацию по конкретному правилу

## Как работает публикация

Общая схема работы:

1. Папки с контентом монтируются в контейнер через Docker volumes, обычно внутрь `/content/...`.
2. В веб-интерфейсе создаются источники, указывающие на эти пути.
3. Сканер индексирует файлы в SQLite, не копируя сами файлы в базу.
4. Каналы хранят Telegram-настройки: токен, `chat_id`, подпись по умолчанию, режим разметки и флаги отправки.
5. Правила выбирают файлы из проиндексированного набора и отправляют их в Telegram по расписанию.
6. Каждая попытка записывается в историю с результатом.
7. Превью генерируются и кешируются на диске отдельно от базы.

Важно: оригинальные медиафайлы остаются в подключённых папках. База хранит только метаданные, индекс и историю.

## Источники контента

Источник хранит:

- имя
- путь внутри контейнера
- режим рекурсивного сканирования
- выбранные типы файлов
- интервал фонового сканирования
- флаг `Только ручное сканирование`

Поддерживаются два режима запуска сканирования:

- `Полный рескан`
- `Добавить отсутствующие`

Во время сканирования интерфейс показывает прогресс.

Вложенные папки поддерживаются. Источник может указывать как на корень volume, так и на вложенную директорию внутри него.

## Поддерживаемые типы файлов

В интерфейсе используются четыре категории:

Поддерживаемые фото-расширения:

## Превью файлов

Сейчас поддерживаются:

- изображения — обычная миниатюра
- GIF — статичное превью по первому кадру
- видео — кадр примерно с 10% длины ролика, но в пределах от 2 до 5 секунд
- документы — иконка-заглушка

Оригинал открывается по клику из интерфейса.

Кеш превью хранится в:

- `/data/previews`

## Конфигурация

Основные переменные окружения:

| Переменная | Значение по умолчанию | Назначение |
| --- | --- | --- |
| `APP_DATA_DIR` | `./data` или `/data` в контейнере | Служебные данные приложения |
| `APP_DB_PATH` | `APP_DATA_DIR/app.db` | Путь к SQLite-базе |
| `APP_TIMEZONE` | `UTC` | Часовой пояс приложения |
| `SCHEDULER_TICK_SECONDS` | `30` | Частота проверки планировщика |
| `SCAN_TICK_SECONDS` | `120` | Частота фонового обхода источников |
| `MAX_CAPTION_LENGTH` | `1024` | Максимальная длина подписи |
| `PREVIEW_BATCH_SIZE` | `24` | Размер партии фоновой генерации превью |
| `APP_SESSION_SECRET` | пусто | Секрет для подписи session cookie |
| `APP_SESSION_HTTPS_ONLY` | `false` | Включает флаг `Secure` для cookie при работе за HTTPS |

Если `APP_SESSION_SECRET` не задан, приложение автоматически создаёт секрет и сохраняет его в:

- `/data/.session_secret`

## Volumes и хранение данных

Обычно используются три типа путей:

- `/data`
  - кеши
  - превью
  - служебные файлы приложения
  - автоматически созданный `session secret`, если он не задан явно

- `/db`
  - SQLite-база `app.db`

- `/content/...`
  - папки с оригинальными медиафайлами

Пример:

```yaml
volumes:
  - ./data:/data
  - ./db:/db
  - ./example-content:/content/example:ro
```

Можно подключать несколько папок:

```yaml
volumes:
  - ./data:/data
  - ./db:/db
  - /srv/media/photos:/content/photos:ro
  - /srv/media/videos:/content/videos:ro
  - /srv/media/memes:/content/memes:ro
```

Тогда в интерфейсе можно использовать пути:

- `/content/photos`
- `/content/videos`
- `/content/memes`

## Как работает выбор файлов

На уровне правила сейчас доступны режимы выбора файлов:

- `Случайно без повторов`
- `Перемешанный цикл`
- `По очереди от старых`
- `Полный случайный выбор`

Одно правило может использовать несколько источников. В текущей версии используется режим выбора источников:

- `Общий пул`

Это означает, что файлы из всех источников правила объединяются в один набор, а дальше к нему применяется выбранный режим выбора файлов.

## Как работает расписание

У правила есть несколько независимых параметров:

- `Интервал` — базовый интервал между событиями публикации
- `Рандомизация времени` — случайное отклонение от базового интервала
- `Публиковать с часа / до часа` — окно активности
- `Постов за событие` — сколько публикаций делать в одной серии
- `Интервал между постами` — шаг внутри серии

Пример:

- интервал `100` минут
- постов за событие `5`
- интервал между постами `2`

Результат:

- каждые 100 минут стартует новое событие
- внутри него публикуются 5 файлов с шагом 2 минуты

## Работа с каналами

Каждый канал хранит:

- название
- `Chat ID / канал`
- токен Telegram-бота
- `Parse mode`
- подпись по умолчанию
- `disable_notification`
- `protect_content`
- флаг включения

У каждого канала есть отдельные страницы:

- `Обзор`
- `Правила`
- `Источники`
- `История`

## История и очередь

История умеет:

- фильтр по статусу
- фильтр по каналу
- фильтр по правилу
- фильтр по источнику
- показ миниатюры файла
- открытие оригинала

Статусы в интерфейсе:

- `Отправлено`
- `Ошибка`
- `Пропущено`
- `Отправлено вручную`

В очереди правила можно:

- посмотреть ближайших кандидатов
- открыть оригинал по клику
- отправить конкретный файл вручную кнопкой `Запостить сейчас`

Ручная отправка фиксируется в истории отдельно от обычного запуска правила.

## HEIC / HEIF

Если файл фактически является `HEIC/HEIF`, приложение может работать с ним по двум сценариям:

1. Отправить как документ.
2. Временно конвертировать в JPEG и отправить как изображение.

Этот режим настраивается на уровне правила.

- один администратор

## Безопасность

Минимальные рекомендации перед использованием вне локальной сети:

- сразу сменить стандартный логин и пароль администратора
- явно задать `APP_SESSION_SECRET`
- использовать HTTPS через reverse proxy
- включить `APP_SESSION_HTTPS_ONLY=true` при работе за HTTPS
- не открывать панель напрямую в интернет без дополнительной защиты
- регулярно делать резервные копии базы

## FAQ

### Что писать в `Chat ID / канал`?

Можно использовать:

- публичный username канала, например `@my_channel`
- числовой `chat_id`, например `-1001234567890`

### Как получить `chat_id` для приватного канала?

1. Добавьте бота в канал как администратора.
2. Опубликуйте в канале любое сообщение.
3. Откройте:

```text
https://api.telegram.org/botТОКЕН_БОТА/getUpdates
```

4. Найдите в ответе поле:

```json
"chat": {
  "id": -1001234567890,
  "type": "channel"
}
```

Именно это значение `id` и нужно вписать в поле канала.

### Почему файл не отправился как изображение?

На практике частая причина — файл имеет одно расширение, но другой реальный формат внутри. Например, файл с расширением `.png` может фактически оказаться HEIC-контейнером.

### Почему не видно нужную папку в выборе источника?

В интерфейсе показываются только доступные директории внутри уже смонтированных volumes `/content/...`. Если была добавлена новая папка или вложенная структура изменилась, обновите список папок в интерфейсе.

### Хранятся ли сами файлы в базе?

Нет. База хранит только метаданные, индекс, настройки и историю. Оригинальные файлы остаются в подключённых папках.
